# Безопасность и приватность ZELENIAL

Этот документ описывает правила защиты данных, приложений, инфраструктуры и пользовательской приватности. Если в архитектурном документе вопрос безопасности упоминается кратко, основным источником истины остается именно этот файл.

## 1. Цели

- защищать персональные и чувствительные данные пользователей
- минимизировать сбор и хранение лишней информации
- проектировать безопасность как часть системы, а не как позднюю надстройку
- обеспечить внятные правила для разработки, логирования, интеграций и AI-функций

## 2. Базовые принципы

- Privacy by design.
- Security by default.
- Least privilege.
- Explicit access control.
- Defense in depth.
- Минимизация данных.
- Трассируемость критичных действий без утечки чувствительной информации.

## 3. Классификация данных

| Класс | Примеры | Требования |
|---|---|---|
| Public | публичное описание продукта, статический контент | минимальные ограничения |
| Operational | технические события, health metrics, обезличенные логи | без персональных данных |
| Personal | email, профиль, настройки, устройства | ограниченный доступ, аудит, защита в storage и transport |
| Sensitive | данные о здоровье, финансах, рефлексии, токены, коды 2FA | повышенная защита, отдельные правила обработки и логирования |

## 4. Сбор и хранение данных

- Собирать только те данные, без которых нельзя выполнить продуктовую задачу.
- Любое новое поле должно иметь понятное обоснование.
- Чувствительные категории данных не должны использоваться в аналитике без отдельной анонимизации.
- Персистентные модели должны проектироваться с учетом жизненного цикла данных: создание, обновление, удаление, экспорт, обезличивание.

## 5. Транспорт и хранение

- Все внешние соединения работают только по HTTPS/TLS.
- Секреты не хранятся в репозитории.
- Пароли и секретные коды не сохраняются в открытом виде.
- Чувствительные поля защищаются на уровне storage и operational access.
- Бэкапы должны быть защищены не хуже основного хранилища.

## 6. Аутентификация и авторизация

Базовые требования:

- безопасная регистрация и вход
- управление сессиями
- поддержка 2FA для чувствительных сценариев
- доверенные устройства и история сессий
- отозвание сессий и ограничение риска при компрометации

Правила:

- аутентификация и авторизация — это отдельный доменный блок, а не набор middleware-хака
- проверки доступа выполняются явно
- права не должны прятаться внутри UI или контроллеров
- чувствительные операции требуют дополнительной проверки при необходимости

## 7. Логи, аудит и наблюдаемость

- Логи не должны содержать пароли, токены, коды 2FA и чувствительные пользовательские записи.
- Для расследования инцидентов должны сохраняться безопасные audit-события.
- Correlation IDs обязательны для трассировки сквозных операций.
- Debug logging не должен попадать в production без контроля.

## 8. Интеграции и внешние сервисы

- Каждая интеграция проходит через явный outbound port.
- У каждой интеграции должен быть владелец, контракт и список передаваемых данных.
- Нельзя отправлять избыточные персональные данные во внешние сервисы.
- Перед подключением нового провайдера оцениваются риск, юридическая применимость и необходимость.

## 9. AI и работа с пользовательскими данными

- AI-функции не получают доступ к данным по умолчанию.
- Передача данных в модели должна быть минимальной и оправданной.
- При возможности используется обезличивание или редукция контекста.
- Нельзя отправлять чувствительные записи в сторонние модели без отдельного архитектурного решения.
- Пользователь должен понимать, где его данные анализируются автоматически.

## 10. Frontend security

- Не хранить чувствительные секреты в клиентском коде.
- Минимизировать объем данных, доступных браузеру.
- Все клиентские проверки считаются вспомогательными, а не доверенными.
- Платформенные адаптеры Capacitor/Tauri не должны размывать модель разрешений.

## 11. Secure development lifecycle

- Любое изменение в доменной модели, API или хранении данных оценивается с точки зрения безопасности.
- Зависимости и обновления должны регулярно проверяться.
- Новые сценарии входа, интеграции и хранения данных проходят threat review.
- Code review обязан проверять не только корректность, но и риск утечки, эскалации прав и неправильного хранения данных.

## 12. Incident readiness

Минимум, который должен быть обеспечен:

- понятный способ отозвать сессии
- журнал критичных событий
- план действий при компрометации секретов
- резервное копирование и восстановление
- безопасная ротация ключей и токенов
