# Правила работы над проектом ZELENIAL

Этот документ описывает правила разработки, ведения репозитория, pull request, code review, документации и использования ИИ в проекте.

Технические соглашения по архитектуре и code style подробно описаны в [ARCHITECTURE.md](ARCHITECTURE.md). Здесь фиксируются обязательные процессные правила и критерии качества.

## 1. Общие принципы

- Один репозиторий, одна архитектурная модель, один набор правил.
- Любое изменение должно быть осмысленным, локальным по ответственности и объяснимым.
- Документация и структура важны не меньше кода.
- Если вопрос уже раскрыт в профильном документе, не дублировать его в другом месте.

## 2. Правила репозитория

- Корневой `README.md` должен оставаться самодостаточным.
- Подробные технические детали живут в `docs/ARCHITECTURE.md`.
- Безопасность и приватность живут в `docs/SECURITY.md`.
- Процессные правила живут в `docs/RULES.md`.
- Порядок реализации живет в `docs/ROADMAP.md`.

Структура репозитория не должна размываться:

- `backend/` — backend-каркас и реализация монолита
- `frontend/` — единый frontend-кодбейс
- `docs/` — документация проекта

Запрещено:

- создавать параллельные README с дублирующим содержанием без необходимости
- разносить архитектурные правила по случайным markdown-файлам
- хранить временные файлы, IDE-мусор, локальные дампы и артефакты сборки в git

## 3. Ветки, коммиты и история

- `main` — главная ветка проекта.
- Одна ветка — одна логическая задача.
- История коммитов должна быть понятной и читаемой.
- Предпочтительный стиль коммитов: Conventional Commits.

Примеры:

- `feat: add planning contracts`
- `fix: align auth flow with domain contracts`
- `docs: rewrite architecture and roadmap`
- `chore: initialize repository structure`

## 4. Pull Request

Каждый PR должен быть достаточно маленьким, чтобы его можно было качественно проверить.

PR обязан содержать:

- цель изменения
- причину изменения
- краткое описание области влияния
- список проверок, которые были выполнены
- явное указание на архитектурные или контрактные изменения

Нельзя смешивать в одном PR:

- крупный рефакторинг и новую фичу без необходимости
- изменение инфраструктуры и случайные сопутствующие правки
- форматирование всего проекта ради одной локальной задачи

## 5. Code Review

Цель review — защищать качество проекта, а не формально одобрять изменения.

В review обязательно проверяются:

- соответствие архитектуре
- корректность доменных границ
- чистота зависимостей
- читаемость и простота решений
- безопасность и работа с данными
- тестируемость
- отсутствие скрытого технического долга

Приоритет review:

1. Ошибки в логике и архитектуре.
2. Риски безопасности и целостности данных.
3. Регрессии поведения и слабые контракты.
4. Качество API и читаемость.
5. Стиль и мелкие косметические вопросы.

## 6. Документация

- Документация обновляется вместе с изменением архитектуры, контракта или процесса.
- Если решение меняет базовые допущения проекта, оно должно быть отражено в архитектурном документе.
- Если текст начинает повторяться в двух местах, остается один основной источник, а во втором месте — краткая ссылка.
- Документация должна быть пригодной для чтения новым участником без знания истории чата и устных договоренностей.

## 7. Обязательные технические правила

Полные технические соглашения находятся в [ARCHITECTURE.md](ARCHITECTURE.md), но как обязательный минимум:

- доменная логика не живет в инфраструктурных слоях
- контракты и use cases не смешиваются с transport models
- код должен быть современным, чистым и явно структурированным
- комментарии добавляются только там, где без них теряется смысл решения
- временные заглушки и мертвый код нельзя оставлять молча

## 8. Правила для ИИ-ассистентов

ИИ в этом проекте работает как mentor, reviewer, architect assistant и quality gate, если явно не согласовано иное.

ИИ обязан:

- уважать архитектуру и не предлагать решения, ломающие границы без явного объяснения
- писать чисто, профессионально, современно и без лишнего мусора, если его просят подготовить пример или патч
- объяснять компромиссы и риски, а не просто выдавать фрагменты кода
- комментировать код только там, где комментарий действительно нужен
- не раздувать абстракции и не предлагать паттерны без реальной пользы
- не добавлять магию, скрытые глобальные состояния и сомнительные shortcuts
- явно отмечать, где решение является рекомендацией, а где обязательным правилом проекта

ИИ не должен:

- брать на себя роль основного разработчика по умолчанию
- без запроса производить большие объемы продуктового кода вместо автора проекта
- плодить дублирующие документы, шаблоны и “умный” мусор
- маскировать неопределенность уверенным тоном без проверки

## 9. Definition of Done

Задача считается завершенной, когда:

- результат соответствует архитектуре и правилам проекта
- изменены только нужные файлы
- документация обновлена, если это необходимо
- выполнены релевантные проверки
- изменение можно понять без устных пояснений
