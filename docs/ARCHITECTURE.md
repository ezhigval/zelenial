# Архитектура ZELENIAL

Этот документ — единый технический источник истины по проекту. Он объединяет архитектурное описание, основные решения уровня ADR/RFC, принципы проектирования, стек, технические соглашения и иллюстративные примеры.

## 1. Назначение документа

Документ фиксирует:

- зачем проект устроен именно так
- как делятся ответственность и зависимости
- какие технологии и подходы являются базовыми
- как должен выглядеть каркас backend и frontend
- какие технические правила обязательны при реализации

Вопросы безопасности и приватности здесь упоминаются кратко. Полное описание находится в [SECURITY.md](SECURITY.md).

## 2. Архитектурные решения верхнего уровня

### ADR-001. Вместо микросервисов используем модульный монолит

Причина:

- проект на раннем этапе не выигрывает от распределенной сложности
- доменные блоки еще формируются
- высокая скорость изменений важнее сетевой изоляции
- проще поддерживать транзакционную целостность, отладку и локальную разработку

Следствие:

- все блоки живут в одном backend-приложении
- каждый блок имеет четкую внутреннюю границу
- вынос в отдельный сервис допускается только после доказанной необходимости

### ADR-002. Backend строится по гексагональной архитектуре

Причина:

- бизнес-логика не должна зависеть от HTTP, базы данных и внешних SDK
- тестирование use cases должно быть быстрым и изолированным
- замена адаптеров не должна ломать доменное ядро

Следствие:

- центр системы — сущности, value objects, доменные сервисы и use cases
- все внешние взаимодействия проходят через порты
- инфраструктура реализует адаптеры, а не управляет бизнес-логикой

### ADR-003. Frontend — единое универсальное веб-приложение

Причина:

- один кодбейс дешевле в поддержке и быстрее в развитии
- продукт должен работать в браузере и устанавливаться на разные устройства
- UI и бизнес-правила должны быть едины на всех платформах

Следствие:

- основная платформа — web + PWA
- мобильные сборки — через Capacitor
- десктопные сборки — через Tauri
- платформенные различия изолируются в тонких адаптерах

### ADR-004. Взаимодействие между блоками только через контракты

Причина:

- прямые вызовы между внутренними пакетами быстро размывают границы
- контрактные модели упрощают тестирование и будущий рефакторинг

Следствие:

- один блок не должен читать внутренние структуры другого напрямую
- обмен проходит через публичные contracts, порты и сценарии использования
- DTO и transport-модели не подменяют доменные сущности

## 3. Системный контекст

ZELENIAL — это единое приложение для управления жизнью. Внутри монолита планируются следующие функциональные блоки:

| Блок | Ответственность |
|---|---|
| Identity | регистрация, вход, сессии, 2FA, доверенные устройства |
| Profile | профиль, настройки, предпочтения пользователя |
| Planning | календарь, задачи, привычки, напоминания |
| Health | состояние, сон, активность, стресс, самонаблюдение |
| Finance | бюджет, расходы, доходы, регулярные платежи |
| Learning | обучение, навыки, учебные треки |
| Reflection | дневник, заметки, рефлексия, выводы |
| Notifications | доставка системных и пользовательских уведомлений |
| Assistant | рекомендации, персонализация, AI-оркестрация |

Правило границ:

- каждый блок владеет своей бизнес-логикой и своими контрактами
- пересечение возможно только через явно описанные зависимости
- общие утилиты не должны превращаться в глобальный мусорный слой

## 4. Backend: модель гексагональной архитектуры

### 4.1. Каркас директорий

```text
backend/
├── entities/                  # доменные сущности
├── values/                    # value objects и типы предметной области
├── contracts/                 # публичные контракты между блоками и transport-модели
│   ├── commands/
│   ├── queries/
│   └── dto/
├── cases/                     # application/use cases
├── ports/
│   ├── inbound/               # входные контракты сценариев
│   └── outbound/              # зависимости на БД, очереди, внешние API, clock, id generators
├── adapters/
│   ├── inbound/
│   │   └── http/
│   └── outbound/
│       ├── persistence/
│       ├── integrations/
│       └── messaging/
├── infrastructure/
│   ├── config/
│   ├── migrations/
│   └── bootstrap/
└── tests/
    ├── unit/
    └── integration/
```

### 4.2. Смысл слоев

| Слой | Что содержит | От чего не зависит |
|---|---|---|
| `entities` | главные сущности и инварианты | HTTP, SQL, JSON, framework types |
| `values` | типы предметной области, правила валидации значений | инфраструктура |
| `contracts` | стабильные команды, запросы и DTO между блоками | конкретные адаптеры |
| `cases` | сценарии использования и оркестрация бизнес-операций | web framework, SQL driver |
| `ports` | интерфейсы входа и выхода | конкретные реализации |
| `adapters` | HTTP, persistence, внешние API, очереди | могут зависеть от библиотек |
| `infrastructure` | конфигурация, wiring, миграции, запуск | может знать о runtime и среде |

### 4.3. Правила зависимостей

- `entities` и `values` не импортируют ничего, что относится к инфраструктуре.
- `cases` могут зависеть только от домена, контрактов и портов.
- `ports` определяются рядом с потребителем зависимости, а не рядом с реализацией.
- `adapters` реализуют порты и преобразуют transport/input/output модели.
- `infrastructure` собирает приложение, но не хранит бизнес-логику.

Запрещено:

- протаскивать `http.Request`, `gin.Context`, SQL rows или ORM models в домен
- хранить бизнес-правила в хендлерах, репозиториях и middleware
- обращаться к БД из `entities` или `cases` напрямую, минуя `ports.outbound`

### 4.4. Поток выполнения запроса

Нормальный путь обработки:

1. Входной адаптер принимает внешний запрос.
2. Адаптер валидирует transport-формат и маппит его в `contracts`.
3. Use case выполняет сценарий и обращается к `ports.outbound`.
4. Outbound adapters работают с БД, кэшем, внешними API.
5. Результат возвращается обратно через `contracts/dto`.
6. Входной адаптер сериализует ответ наружу.

### 4.5. Доменные границы и внутренние контракты

Внутри одного монолита допустимы вызовы между блоками, но только так:

- через `contracts` вызываемого блока
- через `ports.inbound`, если нужен формализованный сценарий
- через доменные события или application events, если нужен асинхронный отклик

Не допускается:

- импорт внутренних пакетов соседнего блока
- совместное владение одной и той же сущностью из разных блоков
- смешивание таблиц и структур без явного контракта

### 4.6. Данные и персистентность

Базовая стратегия:

- одна PostgreSQL-база на весь монолит
- разграничение по схемам, таблицам и ownership на уровне доменных блоков
- миграции версионируются и проходят через `backend/infrastructure/migrations`
- SQL и модели хранения не считаются доменной моделью

Рекомендуемый подход:

- писать SQL явно и контролируемо
- использовать `pgx` как базовый драйвер PostgreSQL
- применять `sqlc` или аналог только если он не ломает границы модели
- не вводить ORM как источник скрытой магии

### 4.7. Внутренние события и будущая эволюция

Монолит не запрещает событийную модель внутри себя:

- application events можно использовать для уведомлений, логирования, интеграций и фоновых задач
- если позже появится необходимость вынести блок наружу, внутренние контракты уже будут подготовлены
- premature split в микросервисы не является целью

### 4.8. API и transport layer

На старте принимается такой стандарт:

- внешний API: REST/JSON
- спецификация API: OpenAPI
- версионирование: `/api/v1/...`
- идемпотентность и контракты фиксируются в `contracts`

Если позже появится GraphQL, gRPC или message-driven интеграция, они добавляются как новые адаптеры, а не как замена доменной модели.

### 4.9. Наблюдаемость и операционные аспекты

Кратко:

- структурные логи
- correlation/request ids
- метрики и health endpoints
- трассировка только там, где она реально помогает

Требования к безопасности логов, секретов и данным описаны в [SECURITY.md](SECURITY.md).

## 5. Frontend: единое кроссплатформенное веб-приложение

### 5.1. Цель

Frontend должен существовать как один продуктовый интерфейс, а не как набор несвязанных клиентов.

### 5.2. Модель поставки

| Канал | Способ доставки |
|---|---|
| Browser | обычное веб-приложение |
| PWA | установка из браузера |
| iOS / Android | упаковка через Capacitor |
| macOS / Windows | упаковка через Tauri |

### 5.3. Каркас директорий

```text
frontend/
├── app/                 # bootstrap, routing, providers, composition root
├── processes/           # длинные пользовательские сценарии
├── pages/               # страницы
├── widgets/             # крупные UI-блоки
├── features/            # пользовательские действия
├── entities/            # фронтовые бизнес-сущности и presentation models
├── shared/
│   ├── ui/
│   ├── lib/
│   ├── api/
│   └── config/
├── platform/
│   ├── pwa/
│   ├── capacitor/
│   └── tauri/
└── public/
```

### 5.4. Правила frontend-архитектуры

- UI и platform wrappers не должны содержать скрытую бизнес-логику.
- Работа с API централизуется в `shared/api`.
- Платформенные особенности изолируются в `frontend/platform`.
- Структура интерфейса должна быть модульной и предсказуемой.
- Состояние делится на server state, app state и local UI state.

### 5.5. Базовый frontend stack

- React 19
- TypeScript
- Vite
- TanStack Router
- TanStack Query
- Zod для схем и runtime validation
- TailwindCSS + design tokens через CSS variables
- Vitest и Playwright для тестов

Если в конкретной задаче потребуется отступление от этого набора, изменение должно быть осознанным и задокументированным.

## 8. Технические соглашения по коду

Эта секция является основным источником технического code style. Процессные правила и правила review описаны в [RULES.md](RULES.md).

### 8.1. Общие принципы

- Код должен быть простым для чтения без лишней магии.
- Каждый пакет должен иметь одну понятную ответственность.
- Предпочтение отдается явным зависимостям и явным контрактам.
- Маленькие функции и маленькие файлы лучше огромных многоцелевых файлов.
- Комментарии добавляются только там, где без них теряется смысл решения.

### 8.2. Backend style

- `context.Context` передается первым аргументом во все операции ввода-вывода.
- Ошибки обрабатываются явно.
- Порты размещаются рядом с потребителем зависимости.
- Доменные сущности не знают о JSON, SQL, HTTP, логгерах и окружении.
- Конструкторы должны явно принимать зависимости.
- Глобальное состояние и скрытые singleton-паттерны не допускаются.
- DTO, transport models и persistence models должны быть отделены друг от друга.

### 8.3. Frontend style

- Компоненты должны быть небольшими и одноцелевыми.
- Server state нельзя смешивать с локальным UI state без необходимости.
- Платформенный код не должен расползаться по фичам и страницам.
- Стили опираются на design tokens и общую систему переменных.
- Перегруженные универсальные компоненты хуже нескольких узких и понятных.

### 8.4. Тесты

- Unit tests проверяют доменные инварианты и use cases.
- Integration tests проверяют адаптеры и реальные взаимодействия с инфраструктурой.
- Контрактные тесты обязательны для ключевых входных и выходных границ.
- Тесты должны описывать поведение, а не внутреннюю реализацию.

## 9. Безопасность в контексте архитектуры

Архитектурный минимум:

- аутентификация и сессии проектируются как отдельный доменный блок
- чувствительные данные не должны попадать в обычные логи
- доступы и права должны проходить через явные проверки
- все transport и storage решения обязаны соответствовать [SECURITY.md](SECURITY.md)

## 10. Что не делаем

- не дробим систему на микросервисы заранее
- не делаем отдельные frontend-приложения под каждую платформу
- не смешиваем домен и инфраструктуру
- не строим общие god-packages с хаотичными helper-функциями
- не переносим правила из документа в код частично и выборочно
